# 读者审查报告格式

## 报告结构

读者审查报告是一组发现，每个发现描述一处"出戏"的瞬间。如果没有发现问题，返回空列表。

```yaml
reader_findings:
  - location: "位置描述 + 引用原文片段"
    feeling: "读者此刻的感受或困惑"
    expectation: "读者期待看到什么"
    absence: "文本实际缺少了什么"
```

## 字段说明

| 字段 | 必需 | 用途 |
|------|------|------|
| `location` | 是 | 问题出现的位置。包含章节/段落标记 + 引用原文片段（至少一整句），让 orchestrator 无需回查 story.md |
| `feeling` | 是 | 读者在这个位置的真实感受。用第一人称："我感到困惑"、"我在想这个人怎么了"、"我开始走神" |
| `expectation` | 是 | 读者此刻期待看到什么。"我以为接下来会交代老汉的安危"、"我期待这个线索在后文出现" |
| `absence` | 是 | 文本实际缺失了什么导致了这个出戏。"战斗过程中完全没有提及老汉"、"这个人物在第三节之后再也没出现" |

## 示例

```yaml
reader_findings:
  - location: >
      第一节，杨过与三人战斗段落。原文：「青衣人喝道：'起丝！'四下银丝骤紧……」
      此后至「那卖鱼老汉这才'啊哟'一声」之间约 120 行
    feeling: >
      我一直在想那个老汉怎么样了。他手腕被丝勒住，旁边在打斗，飞刀暗针满天飞，
      可叙述完全忘了他的存在。
    expectation: >
      我期待战斗过程中至少有一笔带过老汉的状态——哪怕杨过用身体挡住他，
      或者老汉趴在地上瑟瑟发抖。
    absence: >
      从老汉手腕僵住到战斗结束，约 120 行内没有任何关于老汉的描写。
      一个大活人在镜头里消失了。
```

## 原则

- **feeling 用第一人称** — 这是读者的主观感受，不是客观分析
- **expectation 和 absence 分开写** — 前者是"我想看到什么"，后者是"文本缺了什么"，两者未必完全对应
- **不给修改建议** — 读者只报告问题，怎么修是 orchestrator 的事
- **引用必须具体** — 包含完整句子和足够上下文
