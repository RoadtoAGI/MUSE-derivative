# Phase 1: 世界构建

## 核心原则

世界规则和设定的每一个选择都应服务于核心价值的冲突——无法产生冲突的设定元素不值得保留。小世界深挖掘、内在或然性法则是麦基两条铁律（原文及案例见 `references/mckee-setting.md`）。

## 输入契约

从 Phase 0 接收：
- `genre` — 用于确定类型惯例
- `core_value` — 用于设计世界规则（世界应能让核心价值冲突自然发生）
- `premise` — 用于锚定设定方向
- `reference_materials`（可选） — 用户提供的背景素材摘要，应融入 `domain_knowledge`

## 执行步骤

### 1. 设定四维

四维：时代、期限、地点、冲突层面（原文及案例见 `references/mckee-setting.md §设定四维`）。

**时代**：故事在历史中的位置（具体年代或架空设定），决定科技水平、社会形态、文化语境。

**期限**：故事的时间跨度。短（数小时，如《十二怒汉》）意味着高压集中；长（数年/数十年）意味着成长变迁。

**地点**：故事的空间范围。单一封闭空间（如法庭、潜艇）天然制造压力；广阔空间需要通过其他手段集中冲突。

**冲突层面**：故事涉及的冲突维度，按世界特点自由列出。麦基将冲突分为内在（人物 vs 自我）、个人（人物 vs 人物）、外在（人物 vs 社会/环境）三层，仅供参考。

### 2. 建立类型惯例

根据 Phase 0 选定的类型，列出该类型的核心惯例——观众/读者期待看到的元素。惯例是类型合同中的必要条款，不是陈词滥调（原文见 `references/mckee-setting.md §类型惯例`）。

同时标注计划的惯例颠覆点：在哪些地方有意打破惯例以注入原创性。

### 3. 选择类型的冲突生成机制

> 「故事必须遵守其自身内在的或然性法则。因此，作家的事件选择局限于他所创造的世界内的可能性和或然性。」
> —— 《故事》第三章

或然性法则要先有**因**才有**果**。直接写"基础设施崩溃、信息断供、秩序瓦解、物资匮乏"这类只是后果的描述，等于把果当因——下游 Phase 5/6 无法据此推导事件、场景细节和人物行动后果，只能临场发明，破坏世界一致性。

**做法**：消费 Phase 0 的 `genre.primary`，先选定本类型的**冲突生成机制（generative_driver）**，再据此推导世界规则。

题材 → 应优先确定的生成机制（参考对照，非穷举）：

| 题材 | 生成机制类型 | 示例选项（不限于此） |
|------|------------|---------------------|
| 末世 | 灾变成因 + 传播链 | 陨石孢子感染 / 超级真菌污染 / 极寒天灾 / 核尘暴废土 / AI 城市系统失控 / 认知污染 / 异种入侵 / 太阳耀斑电磁灾变 |
| 悬疑 | 隐藏事件 + 线索泄漏方式 | 作案动机与手法、关键证物的扩散路径、谁知道什么、信息何时暴露 |
| 仙侠 | 灵气来源 + 修炼规则 + 资源垄断结构 | 灵气分布、修为阶梯、宗门/家族对资源的控制 |
| 宫斗 | 权力结构 + 继承规则 + 信息流动 | 后位/储位继承法、内廷外朝信息差、宦官/外戚的信道 |
| 科幻 | 技术假设/物理公理 + 运作机制 | 公理层（物理定律变更）+ 机制层（基于公理的具体应用），见 `references/mckee-setting.md §生成机制 vs 结果状态`。社会后果（阶级分化、技术伦理崩坏等）归 `world_rules.social`，不进 driver |
| 都市/爽文 | 身份差/资源差 + 阶层跃迁机制 | 主角的非常规资源来源、阶层壁垒的具体形态 |
| 言情 | 关系阻力 + 误会/身份限制机制 | 阻力的结构性来源（家族、阶级、身份、契约）、误会触发条件 |

**原因 vs 后果的判别**：

- 一个描述能回答"为什么会这样 / 如何扩散 / 触发什么进一步事件"——是**原因**，可作 generative_driver
- 一个描述只回答"现在是什么样"——是**后果**，写入 `world_rules.social/physical` 但不能冒充 driver

末世题材的反例（**禁止**作 driver 主类型）：
- "城市基础设施崩溃型末世"
- "低信息断供末世"
- "秩序瓦解型末世"
- "物资匮乏型末世"

这些是任何一种末世都会出现的共同后果，不构成区分性原因，也不能派生场景的具体威胁、可用资源、行动逻辑。

**输出**：在 `generative_driver` 字段（自由结构）记录本题材的机制选择，至少回答清楚"原因 / 扩散或运作方式 / 在本故事局部场景的具体表现"。子字段命名不强制——按题材自然组织即可。

**单题材实证参考**：本节对照表给方向，每题材的 driver 选型示例 + 名著锚点 few-shot + 平台样态 + 高频禁区单独承载于 [`references/genre-worldbuilding/<slug>.md`](references/genre-worldbuilding/README.md)（apocalypse / mystery / xianxia / palace-intrigue / scifi / romance / wuxia）。按 Phase 0 `genre.primary` 映射 slug 按需加载——不要预读整个目录。混合题材按主导 driver 选一份主文件。

### 4. 由生成机制推导世界规则

> 「世界越小，作者的知识便越完善，其创作选择也就越多。」
> —— 《故事》第三章

世界规则分三个层面，**每条规则应能追溯到 `generative_driver`**：

- **物理规则**：这个世界的自然法则（现实/魔法/科幻？特殊能力或限制？）
- **社会规则**：权力结构、阶级、文化禁忌、仪式、经济运作
- **心理规则**：人物心理是否遵循真实人性？有无特殊心理设定？

**关键**：
- 世界规则必须服务于核心价值的冲突。如果核心价值是"自由 vs 束缚"，世界规则应该自然产生束缚的力量
- 灾后/事后的状态描述（基础设施、信息、秩序、物资）写在 `world_rules.social` 或 `world_rules.physical` 里，是 driver 的派生，不与 driver 并列
- 不要同时声明多个互斥的 generative_driver（例：末世既是孢子感染又是太阳耀斑）。混合题材也只能有一个主导 driver，其它降为辅助层

### 5. 世界研究：日常生活的质感（按需）

> 「我的人物靠什么为生？……我的世界的政治是什么？……我的世界的仪式是什么？……我的世界的价值观是什么？」
> —— 《故事》第八章

世界框架（四维 + 规则）是骨架，日常生活的质感是血肉。没有血肉的骨架会导致 Phase 6 写作时细节泛化。

**触发条件**：当故事世界远离当代日常经验时——历史题材、架空世界、异文化背景、未来设定、特殊亚文化——必须执行本步骤。当代现实背景（如现代都市、读者熟悉的职场/校园）可跳过，因为模型和读者共享足够的常识。判断标准：**如果一个普通读者无法凭日常经验想象出这个世界里"买一碗饭"的完整过程，就需要世界研究。**

**执行方式**：使用 Agent tool 调度 `world-researcher` subagent（已注册 agent，无需读取定义文件）。**不要跳过这一步自己写 daily_life——必须通过 subagent 联网调研获取真实资料。**

具体操作：
1. 使用 Agent tool，`subagent_type` 设为 `world-researcher`，prompt 中传入下方上下文（含输出路径）
2. 等待 subagent 完成，subagent 自行将报告写入指定路径
3. 审阅报告，将关键发现填充到 `daily_life` 字段

传入 subagent 的上下文（作为 Agent tool 的 prompt）：
- **输出文件路径**：`{pipeline目录路径}/world_research.md`——subagent 必须将完整调研报告写入此路径，**不得**写到任何其他位置（如 `results/world-research/`、项目根目录等）
- Phase 0 的 premise
- 步骤 1-4 的产出（时代、地点、generative_driver、世界规则摘要）
- 需要调研的具体问题（根据故事需要指定侧重方向）
- Phase 0 的 `reference_materials`（如有）——用户已提供的素材不需要重复调研，subagent 应聚焦于用户素材未覆盖的维度

**注意**：不需要写百科全书。聚焦于与故事冲突相关的细节——能产生压力、选择和冲突的日常细节才值得保留。

**叙事功能要求**：世界细节不是装饰。保留一个日常细节前必须回答它承担哪种叙事功能：

- 改变角色选择
- 制造 / 修正误读
- 暴露阶层、时代、职业或权力关系
- 把宏大压力落到可感知物件
- 触发行动交换

不能回答以上任一功能的细节，删除——它是装饰，不是世界质感。

**名著示例**：

- 《射雕英雄传》开场冬夜酒馆说书 — 历史背景嵌入生活场景，不靠作者铺陈（scene_23 lines 3-12）
- 《三体Ⅱ》三个老人闲聊 — 末日哲学藏在粮票 / 太空葬 / 废矿井（scene_S18 lines 3-15）
- 《射雕》黄蓉以厨艺换郭靖学掌法 — 食物是情节引擎，不是风味装饰（scene_17 lines 3-12）

### 6. 识别创造性局限

明确这个世界里"什么不可能"。这些限制看似约束，实则是冲突的来源和故事独特性的保障。一个没有限制的世界无法产生有意义的选择。

## 输出

→ YAML 输出结构见 `references/output-schema.md`

交付物写入 `pipeline/phase1_world.yaml`，包含：setting(4D), genre_conventions, generative_driver, world_rules, daily_life(维度灵活，常见维度：livelihood, rituals, values, power_dynamics), domain_knowledge(可选), creative_constraints。如执行了世界调研，完整报告另存 `pipeline/world_research.md`。

`domain_knowledge`：承接用户在 Phase 0 提供的 `reference_materials` 中与世界设定相关的专业知识（如书法技法、军事调度理论、文学奖流程等）。与 `daily_life`（agent 调研的日常细节）互补。用户显式提供的素材应被优先尊重。无用户素材时不生成此字段。

## 常见错误

| 错误 | 后果 | 修正 |
|------|------|------|
| 用结果状态冒充生成机制（如末世类型写"基础设施崩溃"） | 下游无法推导事件、场景威胁、人物行动后果，靠临场发明 | 回到步骤 3，按题材选定具体生成机制（末世选具体灾变成因，悬疑选作案/泄漏机制等） |
| 跳过步骤 3 直接堆世界规则细节 | 规则之间没有共同的因果根，互相打架；增删一条没有依据 | world_rules 每条应能追溯到 `generative_driver`；不能追溯的或删或改 |
| 同时声明多个互斥的 generative_driver | 设定自相矛盾，下游不知道按哪个推演 | 选定一个主导 driver；混合题材给主导 + 辅助层级 |
| 世界规则内部矛盾 | 读者发现逻辑漏洞，信任崩塌 | 规则之间做一致性交叉检查 |
